# RustEngine 后续开发计划分析

## 项目现状

### 规模
- **17 个 crate**, ~71K 行源码 (~84K 含测试/基准/示例)
- README 路线图 0-9 阶段全部标记 ✅ 完成
- 20 个示例 (含完整可玩俄罗斯方块), 23 个集成测试文件, 8 个模糊测试目标, CI/CD 已配置

### 关键发现: 声明完成 vs 实际状态

| 领域 | README 状态 | 实际状态 |
|------|------------|---------|
| 物理引擎 | ✅ | **真正完成** — SAT碰撞、约束求解器、CCD、关节约束 (~4.5K行) |
| 网络系统 | ✅ | **真正完成** — TCP/UDP socket、权威服务器、快照同步、NAT穿越 (~7K行) |
| 脚本系统 | ✅ | **真正完成** — Lua (mlua) + WASM (wasmtime) 双运行时 (~3K行) |
| 编辑器UI | ✅ | **结构完成** — 20+模块, egui绘制代码真实, 但视口渲染占位矩形 |
| 编辑器视口 | ✅ | **占位符** — viewport.rs:263-336 绘制2D彩色矩形, 非真实3D场景 |
| 运行时循环 | ❌ | **不存在** — Play/Pause/Stop UI存在但无实际游戏循环执行 |
| 移动/WASM | ✅ | **无代码** — 仅justfile有Android recipe, 零平台特定代码 |

### 代码证据

**viewport.rs:263-336** — `draw_scene_objects()` 将场景对象绘制为带发光效果的彩色矩形:
```rust
let size = 50.0 * h_scale;
let obj_rect = Rect::from_center_size(Pos2::new(screen_x, screen_y), Vec2::new(size, size));
painter.add(Shape::rect_filled(obj_rect, ...)); // 2D矩形, 非3D渲染
```

**viewport_renderer.rs:60** — 离屏渲染目标存在但未桥接到egui:
```rust
egui_texture_id: Option<egui::TextureId>, // 始终为 None
```

**main.rs:68-139** — 编辑器事件循环仅有egui渲染, 无ECS/物理/音频集成:
```rust
event_loop.run(move |event, elwt| {
    // 仅处理: 窗口事件 + egui帧 + 渲染
    // 无 World::step(), PhysicsWorld::step(), AudioManager::update()
});
```

## 后续路线图 (阶段 10-15)

### 阶段 10 — 引擎可运行 (编辑器视口 + 运行时) ⭐ 最高优先级

**目标**: 让编辑器真正可用 — 视口显示真实3D场景, 按Play运行游戏

#### 10.1 将 wgpu 渲染目标桥接到 egui 视口 (2-3周)
- 在 `ViewportRenderer` 中注册离屏纹理到 egui (`egui_wgpu::Renderer::register_native_texture`)
- 替换 `draw_scene_objects()` 的2D矩形绘制, 改为调用 engine-render 延迟管线渲染到离屏目标
- 将渲染结果作为 `egui::Image` 显示在视口面板
- 支持视口大小变化时重建渲染目标
- **涉及 crate**: `engine-editor`, `engine-render`
- **依赖**: 无 (可立即开始)
- **成功标准**: 编辑器视口显示真实3D网格 + PBR材质 + 光照 + 阴影

#### 10.2 编辑器场景 ↔ 渲染管线桥接 (1-2周)
- 确保 `engine-render` 的渲染图可从编辑器的 wgpu device/queue 调用 (共享 `Arc<Device>`, `Arc<Queue>`)
- 将 ECS 查询结果 (MeshRenderer, Transform, Light, Camera) 接入渲染图提交
- 实现摄像机切换 (编辑器摄像机 vs 游戏摄像机)
- **涉及 crate**: `engine-editor`, `engine-render`, `engine-ecs`
- **依赖**: 10.1
- **成功标准**: 带 MeshRenderer + PBRMaterial 组件的场景对象在视口中渲染; 光照生效; 场景/游戏标签切换摄像机

#### 10.3 Play/Pause/Stop 运行时循环 (2-3周)
- 实现 `EditorRuntime`: 在Play时创建独立 World 副本用于运行时执行
- Play: 复制当前场景状态到运行时 World, 启动游戏循环 (固定时间步), 运行 ECS 系统 (物理/输入/音频/脚本)
- Pause: 冻结运行时循环, 保持状态
- Stop: 销毁运行时 World, 恢复编辑器 World
- "游戏" 视口标签显示运行时摄像机视角
- 处理运行时生成的实体 vs 编辑器放置的实体
- **涉及 crate**: `engine-editor`, `engine-core`, `engine-ecs`, `engine-framework`
- **依赖**: 10.2
- **成功标准**: 按Play → 游戏运行 (物理/输入/音频) → 按Pause → 冻结 → 按Stop → 恢复编辑器状态

#### 10.4 运行时调试可视化 (1周)
- 在视口中绘制物理碰撞体线框 (调试可视化)
- 绘制包围盒、速度向量
- 编辑器模式下保持 Gizmo (平移/旋转/缩放), Play模式下禁用
- **涉及 crate**: `engine-editor`, `engine-render`, `engine-physics`
- **依赖**: 10.3
- **成功标准**: 物理碰撞体以线框可见; Play时Gizmo禁用; 调试覆盖层可切换

**阶段 10 成功标准**: 打开编辑器 → 看到3D场景 → 按Play → 游戏运行 → 按Stop → 返回编辑状态。这是最低可用引擎体验。

### 阶段 11 — 生产加固 (4-6周)

**目标**: 使引擎可靠、高性能、经过充分测试

#### 11.1 端到端集成测试 (2周)
- 创建跨 crate 工作流测试:
  - 创建窗口 → 初始化渲染 → 加载glTF → 渲染帧 → 回读像素 → 验证非黑色
  - 创建World → 添加RigidBody+Collider → 物理步进 → 验证位置变化
  - 创建服务器+客户端 → 连接 → 发送快照 → 验证接收
  - 加载场景JSON → 反序列化 → 验证实体数量和组件值
- **涉及 crate**: 全部 (新测试框架)
- **成功标准**: ≥20个端到端测试覆盖完整管线, CI全部通过

#### 11.2 性能分析与优化 (1-2周)
- 在热路径添加 `tracing` 插桩 (ECS查询迭代、渲染提交、物理步进、资源加载)
- 集成 `puffin` 或 `tracy` (feature flag 控制)
- 基准: 10K实体ECS迭代、1000体物理步进、500 draw call延迟渲染、场景加载/序列化
- **涉及 crate**: `engine-ecs`, `engine-physics`, `engine-render`, `engine-core`
- **成功标准**: 发布基准数据; 1000实体+100物理体+200 draw call下无帧超过16ms

#### 11.3 资产管线补全 (1-2周)
- 实现资产导入设置 (纹理压缩格式、网格LOD生成标志、音频采样率), 存储为 `.meta` 文件
- 资产包编译: 将多个资产打包为单一二进制块以快速加载
- 资产数据库: 跟踪哪些资产被哪些场景使用, 检测缺失引用
- **涉及 crate**: `engine-asset`, `engine-editor`
- **成功标准**: 导入设置可在检查器中编辑; 包加载/卸载正常; 缺失资产引用在编辑器中标记

#### 11.4 错误恢复与健壮性 (1周)
- 着色器编译失败优雅降级 (回退到unlit)
- 资产加载失败不崩溃 — 记录警告, 使用占位符
- 渲染设备丢失恢复 (wgpu `DeviceLost` 处理)
- 编辑器自动保存 (定期场景快照)
- **涉及 crate**: `engine-render`, `engine-asset`, `engine-editor`
- **成功标准**: 引擎存活: 缺失着色器文件、损坏资产、GPU驱动崩溃

**阶段 11 成功标准**: 引擎可测试、已分析性能、优雅处理错误。发布性能基准线。

### 阶段 12 — 平台扩展 (8-12周)

**目标**: 在移动端 (Android) 和 Web (WASM) 上运行

#### 12.1 WASM / Web 目标 (3-4周)
- `engine-window`: winit 已支持 web (via web-sys)
- 全局替换 `pollster::block_on` 为异步兼容模式 (WASM不能阻塞)
- `engine-render`: wgpu 已支持 WebGPU/WebGL2 — 验证 feature flags
- `engine-audio`: rodio 不支持WASM — 用 `web-sys::AudioContext` 替换 (feature-gated)
- `engine-asset`: 文件系统加载替换为 `fetch()` API
- `engine-network`: `std::net` TCP/UDP 替换为 WebSocket (feature-gated)
- `engine-physics`: rayon 替换为单线程回退 (WASM单线程)
- **成功标准**: `cargo build --target wasm32-unknown-unknown` 成功; game_flow_demo 在 Chrome 中 ≥30fps

#### 12.2 Android 目标 (3-4周)
- `engine-window`: winit 支持 Android (via android-activity)
- `engine-audio`: rodio 用 ALSA — 需要 OpenSL ES 或 AAudio 后端
- `engine-input`: 扩展触摸事件处理 (多点触控、手势)
- 资产加载: Android APK asset manager 集成
- 构建工具: Gradle wrapper 或 cargo-apk
- **成功标准**: `cargo apk build` 生成可运行APK; game_flow_demo 在Android设备上运行

#### 12.3 移动端特定适配 (2-3周)
- 编辑器不针对移动端 (太复杂) — 但运行时必须工作
- 触摸摄像机控制 (双指缩放、旋转)
- 响应式UI缩放 (engine-ui, egui已支持屏幕密度)
- 方向处理 (横屏/竖屏)
- **成功标准**: 运行时游戏响应触摸输入; UI在手机屏幕上正确缩放

**阶段 12 成功标准**: 引擎在 Windows/macOS/Linux/Android/Web 浏览器上运行。

### 阶段 13 — 内容管线 & 工作流 (4-6周)

**目标**: 专业级资产管线和编辑器工作流

#### 13.1 可视化脚本 (2-3周)
- 扩展现有节点图编辑器, 添加游戏逻辑节点
- 执行流节点: Sequence, Branch, Loop, Delay, Event
- 数据节点: Math, Transform, Input, Physics Raycast, Spawn Entity
- 编译节点图为ECS系统 (每帧解释执行)
- **成功标准**: 仅用可视化脚本创建简单游戏 (如: 输入控制移动物体)

#### 13.2 Prefab 系统 (1-2周)
- Prefab = 可复用场景模板 (实体层级+组件), 存储为资产文件
- 实例化Prefab到场景, 支持覆盖 (修改颜色、位置等)
- Prefab嵌套 (Prefab包含其他Prefab)
- **成功标准**: 创建Prefab → 放置10个实例 → 修改一个 → 恢复Prefab → 变更重置

#### 13.3 协同编辑 (1-2周, 延伸目标)
- 场景文件锁定 / 合并冲突解决
- 通过网络crate实现实时同步 (多用户场景编辑)
- **成功标准**: 两个编辑器实例连接; 一个的变更实时出现在另一个中

**阶段 13 成功标准**: 非程序员可用可视化脚本构建游戏逻辑。Prefab工作流支持模块化关卡设计。

### 阶段 14 — 生态系统 & 可扩展性 (4-6周)

**目标**: 插件系统、Mod支持、社区基础设施

#### 14.1 插件市场基础设施 (2周)
- 插件清单格式 (名称、版本、依赖、入口点、所需引擎版本)
- 插件加载器: 运行时热加载 (通过 `libloading` 动态库加载)
- 本地插件注册表 (已安装插件的JSON数据库)
- **成功标准**: 创建独立插件 → 打包 → 通过编辑器安装 → 引擎加载使用

#### 14.2 Mod 系统 / 运行时脚本扩展 (2周)
- 允许游戏Mod在运行时注册新ECS组件、系统和资产
- 沙盒Mod执行 (通过 engine-script 的 wasmtime)
- Mod加载顺序和依赖解析
- **成功标准**: 加载WASM Mod添加新组件类型和系统 → 游戏无需重编译即可使用

#### 14.3 VR/AR 支持 (2-3周, 延伸目标)
- 集成 OpenXR (via `openxr` crate)
- 立体渲染: 修改渲染图输出左右眼视图
- 头部追踪: HMD姿态映射到摄像机变换
- 控制器输入: VR控制器映射到 engine-input 动作
- **成功标准**: 在VR头显上运行引擎; 看到立体3D场景; 控制器输入正常

**阶段 14 成功标准**: 第三方开发者可通过插件扩展引擎。Modder无需重编译即可添加内容。
